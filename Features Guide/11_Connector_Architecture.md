# Feature: Connector and Tool Architecture

## Overview
Hardcoding Python functions for every LLM agent tool (e.g., `_echo_tool`, `_math_tool`) does not scale for enterprise platforms. **Phase 7** introduces a dynamic **Connector Architecture**. 

Connectors are adapter plugins. Administrators can define a connector (like connecting to a Postgres database or defining a target REST API endpoint). The Connector automatically generates governed MCP tools and registers them into the core `ToolRegistry`. Because they are injected directly into the core registry, these dynamically generated tools automatically inherit all governance features (Taint Tracking, Risk Scoring, Quota Management, and the Approval Engine).

## Architecture

```text
+-----------------------+      +-----------------------+      +-----------------------+
|   Admin Config        |      |   ConnectorManager    |      |   ToolRegistry        |
|   (e.g., DB URL,      +----->+   (Loads & Inits      +----->+   (Core Governance    |
|    API Key, Cwd)      |      |    Connectors)        |      |    Pipeline)          |
+-----------------------+      +-----------+-----------+      +-----------+-----------+
                                           |                              ^
                                           | (injects dynamic tools)      |
                                           v                              |
                               +-----------------------+                  |
                               |   BaseConnector       |                  |
                               |   - RestAPIConnector  +------------------+
                               |   - DatabaseConnector |
                               |   - ShellConnector    |
                               +-----------------------+
```

## How It Works

### 1. The `BaseConnector` Abstraction
Every connector must subclass `BaseConnector` and define:
- `initialize()`: Sets up the connection (e.g., spins up an `httpx` client or an async SQLAlchemy engine).
- `shutdown()`: Cleans up resources.
- `metadata` dictionary: Populates `ConnectorMetadata` defining exactly how the platform should govern the tool. For example, a `sql_execute` tool generated by the `DatabaseConnector` will hardcode `intent_category="write"`, `sensitivity_level="restricted"`, and `is_reversible=False`.

### 2. Available Connectors
- **`RestAPIConnector`**: Takes a `base_url` and an API key. Dynamically generates `http_get` and `http_post` tools. `http_get` is marked as `read` (public), while `http_post` is marked as `write` (internal).
- **`DatabaseConnector`**: Takes a database connection string. Dynamically generates `sql_query` (safe SELECTs) and `sql_execute` (mutations). 
- **`ShellConnector`**: Takes a `sandbox_dir`. Generates a `run_command` tool that strictly enforces execution inside the configured `cwd`. 

### 3. Registry Injection
On startup, `main.py` (via `ToolRegistry.initialize`) registers multiple instances of these connectors. The `ConnectorManager` loops through them, calls `.initialize()`, and then explicitly copies the generated callables and metadata dictionaries into the `ToolRegistry.tools` and `ToolRegistry.tool_metadata` structures.

From the LLM Agent's perspective, it just sees `http_get`, `sql_query`, and `run_command` alongside standard tools like `calculator`. The abstraction is entirely transparent, but fully governed.
